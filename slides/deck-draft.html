The Real Problem

A fintech client of Factored sells charged-off debt. More than a million accounts sold. Each sale generates thousands of affidavits: legal documents signed under oath.

  • Someone must verify that every number in each affidavit matches the operational data.
  • If they do not match, the signer is swearing to false information under oath.
  • Scale: 1M+ accounts sold, ~$60M in proceeds. Thousands of PDFs. Different format per buyer.
Can we automate this with AI? Yes. But should you trust it blindly?

A production data point

First model tried (open-source, Llama 3.1 with 405 billion parameters): missed 22 out of 25 known real exceptions. 88% miss rate.

The model read the PDFs fine and produced valid JSON. It just did not catch the discrepancies. It looked like it was working, but silently passed bad data.

How it fails (concrete example)

The prompt tells the LLM: "The expected chargeoff balance is $12,847.00. Find this value in the document." The document actually says $12,874.00 (transposed digits). The model responds: found: true, value: "$12,847.00". It echoed the expected value instead of reading the document. The Conscience catches this because it compares the extracted value against the operational data independently.

Slide 4 | 0:06 - 0:08 | 2 min

How Document Extraction Evolved

StageApproachStrengthWeakness
1 Rules & Regex Pattern matching on text 100% deterministic, instant Breaks when layout changes
2 OCR Image to text (Tesseract, Textract) Handles scans No semantic understanding
3 Layout-aware parsers Understand tables, columns, reading order Structured extraction Still rule-based per format
4 LLMs / Vision models Read any layout, understand semantics Maximum flexibility Not deterministic. Can hallucinate.

Each leap gains flexibility but loses determinism.

Speaker notes (Spanish):
- "Etapa 1: reglas y regex. Si el formato no cambia, funciona perfecto y es instantáneo."
- "Etapa 2: OCR. Convertir imágenes a texto (Tesseract, Textract)."
- "Etapa 3: parsers que entienden la estructura: tablas, columnas, orden de lectura."
- "Etapa 4: donde estamos hoy. LLMs que leen cualquier formato. Máxima flexibilidad, pero no son deterministas y pueden alucinar."
- "El punto: cada salto ganó flexibilidad y perdió determinismo. Eso importa cuando hay consecuencias legales. Por eso nuestro pipeline usa regex primero y el LLM solo como respaldo."

Si preguntan "¿los LLMs reemplazan a OCR?": "Para documentos complejos y variados, ayudan. Para volumen alto con formato uniforme, no: regex u OCR siguen siendo más baratos, rápidos y deterministas. Lo que gana es el enfoque híbrido: extraer el texto con pypdf/OCR, e interpretar con el LLM solo lo que hace falta."

PDF Types (and Why It Matters)

Identify the type BEFORE choosing the tool.

TypeWhat's insideExtraction
Native / Digital Embedded text (created by software) pypdf extracts perfectly. No AI, no OCR, free, exact.
Scanned / Image Only pixels (photo of a document) Needs OCR or a vision model to "read" the image.
Hybrid Mix: some pages text, some images Detect page by page. Route accordingly.
Interactive forms (AcroForm/XFA) Looks like text, but values live in an internal dictionary Trap: normal parsers see blank fields. Values are invisible to them.
Throwing an LLM at a native-text PDF is like using a microscope to read a billboard.

Speaker notes (Spanish):

- "Juez: humano. Solo revisa los casos marcados REVIEW; los PASS se archivan solos."
- "La frase clave: si el Cerebro alucina, la Conciencia lo atrapa."

Exercise 3: Idempotency

  • Content-hash cache: same PDF + same prompt = skip reprocessing.
  • Upsert by business key: run twice, zero duplicates. Same pattern as MERGE INTO in Snowflake, Databricks, or BigQuery.

15 min

Stretch Wire up process_with_cache() end to end

Speaker notes (Spanish):
- "Idempotencia suena fancy, pero es simple: si corren el pipeline dos veces sobre el mismo PDF, el resultado no se debe duplicar."
- "Sin cache, el LLM reprocesa (cuesta plata y tiempo). Sin upsert, la base duplica filas. Dos mecanismos: un hash del contenido como cache y un upsert con INSERT ON CONFLICT DO UPDATE. Mismo patrón que un MERGE en Snowflake, Databricks o BigQuery."
- "Detalle sutil: el hash incluye la VERSIÓN del prompt. Si cambian el prompt, el cache viejo ya no aplica. Reusar cache tras cambiar el prompt es un bug real que mucha gente ha vivido."
- "Ejercicio 3. Tienen 15 minutos. Core: hash_content() y upsert_result(). Stretch: process_with_cache(). Verifican con pytest tests/test_idempotency.py."

Exercise 4: Orchestration

Why Airflow? A single Python script will not scale. In production, this pipeline pulls PDFs from cloud storage, calls LLMs as external services, writes results to a database, validates against transactional data, and routes REVIEW cases to a human reviewer (the Judge). Airflow orchestrates all of that: retries, dependencies, parallelism per buyer, and visibility into what ran and what failed.
for each buyer (TaskGroup):
  download_pdfs → Eyes + Brain (parse_and_extract) → Conscience (validate_results) → Judge (create_review_tickets)

15 min

Airflow TaskFlow in 30 seconds

@task
def step_a() -> list:     # @task makes it an Airflow task
    return [1, 2, 3]

@task
def step_b(data: list):   # receiving step_a's output = dependency
    print(data)

result = step_a()
step_b(result)             # step_a runs first, step_b waits for it

5 Patterns to Take Home

  1. Eyes first. Identify the PDF type before choosing a tool. Native text? Regex shortcut. Unknown layout? The Brain.
  2. Never trust the Brain without the Conscience. In production: one model missed 88% of exceptions, another caught them all. The model matters, and even then, the Conscience (deterministic guardrails) is non-negotiable.
  3. Hash + upsert = run twice without duplicating. Same pattern in SQLite, Snowflake, or BigQuery. Version the prompt in the cache key.
  4. TaskGroup per entity = clean parallelism in Airflow. The DAG wires Eyes, Brain, Conscience, and Judge per buyer. In production, Airflow orchestrates but does not process.
  5. Sanitize outputs by audience (the Judge sees everything, external parties do not). Full detail to the internal reviewer. Field name + issue type only to external parties. PII-safe by construction.

These patterns are not specific to PDFs. They apply to any pipeline where a model generates data that has real consequences.
